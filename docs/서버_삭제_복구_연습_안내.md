# 서버 삭제·복구 연습 (EC2, 초보용)

EC2에서 “사고가 났다”고 가정하고, Git + Docker로 홈페이지를 되살리는 연습입니다.

---

## 먼저 읽기

| 저장소 | 내용 | Git 복구 | 백업 복구 |
|--------|------|----------|-----------|
| GitHub | 소스 코드 | ✅ | — |
| Docker 볼륨 `postgres_data` | DB 데이터 | ❌ | ✅ `.sql` |
| `.env.ssl`, `ssl-active.conf` | HTTPS | ❌ | WinSCP로 수동 |

**연습 순서:** [DB_백업_복원_안내.md](DB_백업_복원_안내.md)로 **백업 먼저** → HTTPS 설정 PC 백업 → 아래 **2-C**(완전 삭제) 또는 2-A/2-B(같은 서버 내 연습)

---

## 0단계 — 사전 점검

```bash
cd ~/hanwhagreen_test
git pull origin main
chmod +x scripts/ec2-pre-check.sh
./scripts/ec2-pre-check.sh
```

결과 파일: `backups/pre-check_latest.txt`  
복구 후 같은 스크립트를 다시 실행해 **공지·QA 개수**를 비교하세요.

HTTPS 사용 시 WinSCP로 PC에 미리 복사:
- `.env.ssl`
- `nginx/ssl-active.conf`

---

## 2-A — 가벼운 사고 (코드만 삭제, DB 볼륨 유지)

### 상황

프로젝트 폴더를 지웠지만, **`down -v` 는 하지 않음** → DB 데이터는 Docker 볼륨에 남음.

### ① 사고 재현

```bash
cd ~/hanwhagreen_test
docker compose -f docker-compose.yml -f docker-compose.ec2.yml down
cd ~
rm -rf ~/hanwhagreen_test
```

브라우저 `http://퍼블릭IP/` → 접속 불가 확인.

### ② Git 복구

```bash
git clone https://github.com/parkseonguk125/hanwhagreen_test.git ~/hanwhagreen_test
cd ~/hanwhagreen_test
chmod +x scripts/ec2-deploy.sh

# HTTPS 쓰던 경우: .env.ssl, nginx/ssl-active.conf 다시 업로드

./scripts/ec2-deploy.sh
```

### ③ 확인

```bash
./scripts/ec2-pre-check.sh
diff backups/pre-check_latest.txt backups/pre-check_latest.txt
# (복구 전 파일을 다른 이름으로 저장해 두었다면 diff로 비교)
```

- 홈페이지 접속 OK
- **공지·QA 개수가 사고 전과 같으면** 성공 (볼륨이 살아 있었기 때문)

## 2-B — 심각한 사고 (볼륨 삭제 + 백업 복원)

> **반드시** `db-backup.sh`로 백업을 만든 **후** 진행하세요.

### ① 사고 재현

```bash
cd ~/hanwhagreen_test
docker compose -f docker-compose.yml -f docker-compose.ec2.yml down -v
cd ~
rm -rf ~/hanwhagreen_test
```

`-v` = DB 볼륨까지 삭제 → 게시글 데이터 전부 사라짐.

### ② Git으로 코드 복구 (2-A와 동일)

```bash
git clone https://github.com/parkseonguk125/hanwhagreen_test.git ~/hanwhagreen_test
cd ~/hanwhagreen_test
./scripts/ec2-deploy.sh
```

이 시점 DB = **초기 시드만** (공지 1건, QA 1건, admin).

### ③ 백업으로 DB 복구

백업 파일이 clone 전에 `~/` 등 다른 경로에 있어야 합니다.  
**팁:** 2-B 전에 WinSCP로 `backups/*.sql`을 PC에 받아 두고, clone 후 다시 업로드.

```bash
cd ~/hanwhagreen_test
./scripts/db-restore.sh backups/hanwhagreen_YYYYMMDD_HHMMSS.sql
```

### ④ 확인

```bash
./scripts/ec2-pre-check.sh
```

0단계에서 기록한 **공지·QA 개수**가 돌아왔는지 확인.

## 2-C — 최악의 사고 (EC2 인스턴스 완전 삭제 + 전체 복구)

> **반드시** G단계(HTTPS 설정 PC 백업) + `db-backup.sh` + **WinSCP로 `.sql` PC 다운로드** 후 진행하세요.

### 상황

EC2 인스턴스를 **terminate** → 서버 디스크 전체 소멸.  
코드, Docker, DB 볼륨, SSL 인증서, 서버 안 `backups/` **전부 사라짐**.

### ① 종료 전 체크리스트

PC `hanwhagreen-ec2-backup` 폴더에 아래가 있는지 확인:

| 파일 | 필수 |
|------|------|
| `.env` | ✅ |
| `.env.ssl` | ✅ (HTTPS 사용 시) |
| `ssl-active.conf` | ✅ (HTTPS 사용 시) |
| `hanwhagreen_YYYYMMDD_HHMMSS.sql` | ✅ |
| `.pem` / `.ppk` SSH 키 | ✅ |

추가 메모: DuckDNS 도메인명, 보안 그룹(22/80/443) 설정.

### ② 사고 재현 (AWS 콘솔)

1. EC2 → **인스턴스** → 현재 인스턴스 선택
2. **인스턴스 상태** → **인스턴스 종료(terminate)**
3. **terminated** 확인

브라우저 접속 불가 확인.

### ③ 새 EC2 생성

[AWS_EC2_배포_안내.md](AWS_EC2_배포_안내.md)와 동일하게:

- Ubuntu 22.04, 기존 키 페어 재사용
- 보안 그룹: SSH 22, HTTP 80, HTTPS 443
- 새 **퍼블릭 IP** 확인

### ④ DuckDNS IP 갱신

[duckdns.org](https://www.duckdns.org/) → 도메인 IP를 **새 퍼블릭 IP**로 변경.

### ⑤ 서버 초기 설정 (PuTTY)

```bash
git clone https://github.com/parkseonguk125/hanwhagreen_test.git ~/hanwhagreen_test
cd ~/hanwhagreen_test
chmod +x scripts/ec2-setup.sh scripts/ec2-deploy.sh scripts/db-restore.sh
./scripts/ec2-setup.sh
```

`docker` 권한 오류 시 PuTTY 재접속 또는 `newgrp docker`.

### ⑥ PC 백업 업로드 (WinSCP)

| PC 파일 | 서버 위치 |
|---------|-----------|
| `.env` | `~/hanwhagreen_test/` |
| `.env.ssl` | `~/hanwhagreen_test/` |
| `ssl-active.conf` | `~/hanwhagreen_test/nginx/` |
| `hanwhagreen_*.sql` | `~/hanwhagreen_test/backups/` |

### ⑦ 배포 + DB 복원

```bash
cd ~/hanwhagreen_test
./scripts/ec2-deploy.sh
./scripts/db-restore.sh backups/hanwhagreen_YYYYMMDD_HHMMSS.sql
```

### ⑧ HTTPS 재발급

인증서 볼륨도 소멸했으므로 `ssl-init` 필요:

```bash
cd ~/hanwhagreen_test
chmod +x scripts/ssl-init.sh
./scripts/ssl-init.sh
```

### ⑨ 확인

```bash
./scripts/ec2-pre-check.sh
```

0단계 기준선(공지·QA·회원 개수)과 비교. `https://도메인/` 접속 확인.
